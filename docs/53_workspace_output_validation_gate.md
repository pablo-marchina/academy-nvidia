# Workspace Output Validation Gate

## Objective

Epic 26.2 adds a lightweight validation gate for outputs generated by the AI
before a task is considered done. The gate checks structure, contracts, scope,
evidence, uncertainty, and operational checks without changing scoring,
retrieval, recommendation, API behavior, UI behavior, or Action Brief generation.

## What Must Be Validated

Run output validation when a task creates or changes:

- Startup Action Brief JSON or Markdown
- API response JSON
- Answer Quality Eval JSON
- regression dashboard JSON
- generated Markdown docs, prompts, plans, or epics
- code that affects structured outputs

Small hotfixes are intentionally exempt from heavy process. For a 1-5 line
localized fix without structured output changes, record the minimal relevant
check instead of running every output validator.

## Validation Rules

### Action Brief JSON

Use `validate_action_brief_output`.

Checks:

- payload matches `StartupActionBrief`
- required sections are present
- `recommended_motion` is one of the pipeline contract values
- score fields are in the 0-100 range
- gap ids are in `TechnicalGap`
- NVIDIA technologies match the existing mapping matrix for the diagnosed gap
- recommendations with NVIDIA technologies preserve evidence
- `missing_evidence` is present, even when empty
- low confidence includes uncertainty
- critical sections are not empty

### Markdown

Use `validate_markdown_output`.

Checks:

- required headings are present
- critical sections are not empty
- unresolved placeholders such as `TODO`, `TBD`, or `{placeholder}` produce a
  controlled `WARN`

### Regression Dashboard JSON

Use `validate_dashboard_output`.

Checks:

- top-level status is `PASS`, `WARN`, or `FAIL`
- required metric keys are present
- JUnit metrics exist when answer quality JUnit is present
- answer quality status and detail metrics are present when available

### API Response JSON

Use `validate_api_response_output`.

Supported response types:

- `version`
- `rag_status`
- `brief`
- `evaluate`
- `artifacts`

The validator reuses existing FastAPI Pydantic response schemas. Unknown
response types return `WARN`, not a crash. `RagStatusResponse` accepts Qdrant
offline as a controlled state when `qdrant_available=false` and an error is
reported.

## Commands

```bash
make validate-output
make validate-brief-output
make validate-dashboard-output
```

These commands run focused unit tests for the gate. They do not replace:

```bash
pytest
ruff check .
black --check .
mypy src
make validate
```

## Expected AI Behavior

Before closing a non-trivial task, the AI should state what was validated:

- contract/schema
- format
- scope
- evidence and uncertainty preservation
- operational checks run

If a validator returns `FAIL`, fix it if it is in scope. If it is outside scope,
report the failure as a pending item and do not hide it. If there is no schema,
return a controlled `WARN` and document the limitation.

## Non-Goals

- No new external dependencies.
- No mandatory LLM judge.
- No changes to scoring, retrieval, recommendation, API behavior, UI behavior,
  or Action Brief generation.
- No heavy workflow for trivial hotfixes.
